# Voyalier v0.5.1 user-flow reliability implementation plan

Date: 2026-07-21
Status: approved
Release target: `v0.5.1`
Comparison point: `v0.5.0` (`26d8456`)

Source evidence:

- Real Axum + Vite browser audit at desktop, tablet, and 320–375 px widths.
- Keyboard, reload, offline recovery, theme persistence, and planning-form checks.
- Current component, integration, release-workflow, and public-documentation sources.

## Goal

Ship a focused patch release that removes seven verified breaks in otherwise
working trip flows: review-dialog focus and scroll position, stale global health,
mobile topbar overflow, lost focus after review completion, unsynchronized theme
controls, silent blank planning submissions, and lost trip context on reload.
Keep Voyalier local-first, evidence-bounded, accessible, and visually consistent
with the Quiet Journey system.

## Non-goals

- No contract, transport route, storage-schema, provider, parser, booking, sync,
  telemetry, or remote-AI behavior change.
- No change to trip evidence authority, readiness rules, or confirmation review
  semantics.
- No new framework, hosted service, background process, or product claim.
- No OS-signing claim; updater signing remains separate from Apple notarization
  and Windows Authenticode.

## Slice 1 — dialog focus lifecycle

1. Make `Dialog` support a top-context initial focus target and an explicit,
   connected return-focus target.
2. Open candidate review at the title and top of the overlay instead of moving
   the viewport to the first deep **Confirm** button.
3. Return focus to the review trigger while suggestions remain and to the stable
   reviewed-status target when the final candidate removes that trigger.
4. Preserve the existing focus trap, Escape behavior, keyboard queue movement,
   and screen-reader dialog labelling.

Verification: focused component tests plus real-browser assertions for overlay
`scrollTop === 0`, reachable confirmation controls, and a connected focused
element after **Done**.

Commit: `Web: repair review dialog focus flow`

## Slice 2 — coherent engine health

1. Add one web-layer transport-health reporter owned by `App` and available to
   shared async data/action hooks.
2. Normalize gateway failures as before; only `transport/failure` changes the
   shell to offline.
3. Let a successful retry restore the shell to ready so the topbar, banner, and
   local error state cannot contradict each other.
4. Keep the gateway contract and Rust/Axum/Tauri behavior unchanged.

Verification: component tests force an online initial probe, fail an opened-trip
load, assert the global offline state, then recover the same request and assert
ready.

Commit: `Web: synchronize transport health`

## Slice 3 — mobile shell reflow

1. Give the topbar a deliberate two-row layout below 640 px: brand first,
   search/health/theme/settings controls second.
2. Keep every action visible, labelled, keyboard reachable, and at least 44 px in
   target size without document-level horizontal scrolling.
3. Preserve the existing desktop layout and Quiet Journey styling.

Verification: browser checks at 320, 375, 768, and 1440 px; compare
`documentElement.scrollWidth` with `clientWidth`, exercise every control, and
check 200% zoom/reduced-motion behavior.

Commit: `Web: reflow the mobile topbar`

## Slice 4 — shared theme state

1. Replace per-hook theme state with one observable browser-local store.
2. Keep System/Light/Dark persistence and root `data-theme` behavior unchanged.
3. Synchronize simultaneous topbar and Settings controls in the same render
   cycle, including labels and checked states.

Verification: render both controls, switch each direction, and assert the other
control, root theme, and local storage update together; repeat in a browser.

Commit: `Web: share theme preference state`

## Slice 5 — planning validation feedback

1. Disable custom-packing submission until the trimmed label is non-empty and
   mark the input requirement accessibly.
2. Keep manual-item submission available for validation, but move focus to the
   required Name field and show/announce an inline error for blank input.
3. Clear the error as soon as the traveler enters a valid title; preserve core
   validation for all other limits.

Verification: visible-role tests cover disabled packing submission, inline
manual-item feedback, focus, announcement, and successful recovery.

Commit: `Web: explain planning prerequisites`

## Slice 6 — reload re-entry

1. Store only the current trip id in session storage while a trip workspace is
   active; do not persist trip data, evidence, queries, or user content.
2. Restore that trip on reload and keep a valid `#section-*` destination so the
   traveler returns to the same workspace section after data loads.
3. Clear stale section fragments and the session key when returning to the trip
   list; reject malformed persisted values safely.

Verification: unit coverage for safe persistence/clearing plus a real-browser
trip → Plan → reload journey that restores the same title and Plan section.

Commit: `Web: restore trip context on reload`

## Documentation and release synchronization

1. Add a `v0.5.1` changelog entry and dedicated release notes describing only
   verified user-visible outcomes and retained limits.
2. Synchronize version `0.5.1` across root `package.json`, workspace
   `Cargo.toml`, `apps/web/package.json`, and Tauri configuration, then refresh
   lockfiles through their package tools.
3. Update README and user guides only where review focus, offline recovery,
   mobile navigation, theme synchronization, planning validation, or reload
   behavior makes existing prose incomplete.
4. Audit documentation navigation, download links, screenshots, icons, favicons,
   alt text, install commands, unsigned-installer language, and updater guidance.
5. Build and inspect the local docs homepage, download page, planning guide, and
   update guide at desktop/mobile widths and compare them with the deployed site.

Commits: `Docs: document Voyalier 0.5.1` and
`Release: prepare Voyalier 0.5.1`

## Final verification and release

Run focused tests while iterating, then:

```bash
./scripts/check.sh
git diff --check
pnpm audit --prod
```

Also run the credential-string scan from `security-hygiene.yml`, the real Axum +
Vite browser journey, responsive browser QA, docs build/render QA, and the
release workflow's build-only dry run. Refresh Graphify and verify a scoped query
before any tag or publish action.

After every local gate passes:

1. Fetch and integrate any new `origin/main`, rerunning affected checks.
2. Close the release branch with `Merge: Voyalier 0.5.1 user-flow reliability`,
   merge it into `main`, and push `main`.
3. Wait for CI, CodeQL, security hygiene, browser E2E, and Pages on the exact main
   SHA; verify the live docs/download surface.
4. Run and inspect the build-only release workflow, then create and push the
   annotated `v0.5.1` tag.
5. Inspect the protected draft release for macOS DMG/app archive/signature,
   Windows NSIS/MSI/signatures, `latest.json`, platform checksums, and provenance.
6. Verify checksum contents, updater platform entries and signature bindings,
   tag/main alignment, and public HTTP access before publishing the draft.
7. Download the macOS artifact into an isolated location, verify metadata,
   checksum, signature structure, and a representative local-first workflow
   without replacing user data or the installed production app.
8. Verify `releases/latest`, live documentation links, and the final clean tracked
   tree. Keep `packs-v1` prerelease and unchanged.

Release verdict must be `PASS`. The known unsigned/not-notarized installer status
may be reported as an existing limitation, but any functional, accessibility,
artifact, updater, provenance, tag, or exact-SHA gap holds publication.
