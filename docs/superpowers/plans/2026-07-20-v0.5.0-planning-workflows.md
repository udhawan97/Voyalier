## Goal

Ship one coherent minor release that makes Voyalier's existing recommendations
and packing advice durable and actionable, broadens manual planning beyond
flights and stays, makes the local workspace searchable, exposes its complete
data-source contract, expands verified offline maps, and provides a complete
Spanish application interface. Preserve the product's evidence boundaries,
local-first behavior, privacy contract, accessibility, and transport parity.

## Non-goals

- No booking, payment, prices, availability, opening-hours authority, routing,
  live transit, inbox monitoring, collaboration, sync, telemetry, or background
  daemon.
- No parser or AI creation of trip items.
- No translation of imported text, traveler-authored text, government advice,
  pack prose, or model output.
- No automatic promotion of recommendations or packing suggestions into durable
  traveler-owned records.
- No OS code-signing claim. Updater signing remains distinct from Apple
  notarization and Windows Authenticode.

## Confirmed TDD seams

The owner explicitly confirmed these public test seams before tests were written:

1. `voyalier-core` public functions for deterministic validation, ordering,
   overlap detection, safe projections, and workspace-search ranking.
2. `AppService` with temporary SQLite and the shipped fakes for persistence,
   lifecycle, vault, and backup behavior. Feature behavior is read back through
   `AppService`; direct SQL is limited to migration, encryption, and integrity
   guarantees.
3. The versioned `AppGateway`, route manifest, Axum HTTP, and Tauri IPC for wire
   shape and transport parity.
4. React Testing Library through visible roles, labels, focus, announcements,
   and user interactions.
5. Playwright through the real loopback Axum + Vite browser workflow.
6. Pack-builder outputs, `pmtiles verify`, headers, byte limits, and checksums.
7. Packaged Tauri smoke testing for release acceptance.

Each capability is a vertical red → green slice. Do not write a horizontal batch
of imagined tests. Refactoring waits for the final code-review stage.

## Domain and storage shape

Append migration version 10; never edit or reorder versions 2–9.

### `trip_interest_profiles`

- `trip_id` primary/foreign key with cascade delete.
- Five `REAL NOT NULL` weights (`food`, `culture`, `nature`, `nightlife`,
  `shopping`) constrained to 0.0–1.0.
- `updated_at` ISO timestamp.
- No row means Balanced (`0.5` each); reads return that value without writing.

### `saved_places`

- `id`, `trip_id`, `pack_id`, name, category, dimension, latitude, longitude,
  source, license, reasons JSON, wildcard, optional notes, created/updated time.
- Unique identity: trip + pack + folded name + exact coordinates.
- Source-pack removal does not cascade. `sourcePackAvailable` is computed by
  checking the current downloaded-pack relation.
- `notes` is sealed.

### `packing_items`

- `id`, `trip_id`, label, checked, optional suggestion code, created/updated time.
- A partial unique index prevents two items from the same suggestion code in one
  trip while allowing duplicate custom labels deliberately.
- `label` is sealed.

### `trip_items`

- `id`, `trip_id`, kind (`activity | rail | transfer`), title, optional location,
  start/end local wall-clock values, optional notes, optional saved-place id,
  created/updated time.
- Deleting a saved place sets the link to null; deleting the trip cascades.
- `title`, `location`, and `notes` are sealed.

Add every sealed column to `SEALED_COLUMNS` and route all reads/writes through
`Records`. Backup remains a whole-workspace database export, so the new tables
travel automatically; add round-trip and newer-schema fixture coverage.

## Contract shape

Add backward-compatible fields to `TripDetail`:

- `interestProfile`
- `savedPlaces`
- `packingItems`
- `tripItems`

Add types for each record plus inputs. Add these `AppGateway` mutations/read:

- `setInterestProfile(input)`
- `savePlace(input)`
- `updateSavedPlace(input)`
- `deleteSavedPlace(savedPlaceId)`
- `addPackingItem(input)`
- `updatePackingItem(input)`
- `deletePackingItem(packingItemId)`
- `createTripItem(input)`
- `updateTripItem(input)`
- `deleteTripItem(tripItemId)`
- `searchWorkspace(query)`

Every method lands in `AppService`, Axum, Tauri, TypeScript contract/mock, HTTP
gateway, Tauri gateway, and `packages/contracts/parity/routes.json`. Tauri
commands remain snake_case with one `input` argument. Route parity count changes
in both languages in the same slice.

`WorkspaceSearchHit` adds trip id/title/status to the existing labelled result
shape. Source kinds become `document | confirmed_fact | note | saved_place |
trip_item`; pending candidates remain excluded.

## Final verification and review

Focused checks run after every red → green cycle. Typecheck and single test files
run regularly. At the end:

```bash
./scripts/check.sh
git diff --check
pnpm audit --prod
```

Also run the credential-string grep from `security-hygiene.yml`, pack-script
tests, browser E2E, docs build/render QA, and a release workflow dry run.

Run the requested two-axis code review against the committed plan point:

- Standards: `AGENTS.md`, `CONTRIBUTING.md`, architecture rules, and smell baseline.
- Spec: these seven workflow specs plus ADR-0005 and this plan.

Use parallel review agents as required by the code-review skill. Fix every hard
standard violation, missing requirement, incorrect behavior, security/privacy
defect, and material smell before the final merge commit.

## Merge, publish, and shipped proof

1. Fetch current `origin/main`; merge it into the release branch if it advanced,
   preserving concurrent work and rerunning affected gates.
2. Close the branch with `Merge: Voyalier 0.5.0 planning workflows`, merge to
   local `main`, and push `main`.
3. Wait for CI, CodeQL, security hygiene, browser E2E, and docs deployment on the
   exact main SHA. Fix on `main` only when a post-push defect is proven, then
   re-verify the replacement SHA.
4. Run the pack publisher from that SHA and independently download/verify Tokyo
   and Paris descriptors and PMTiles. Confirm Nashville and Kyoto descriptors
   remain present and `packs-v1` stays prerelease.
5. Run the release workflow's build-only dry run. Inspect both platform artifacts.
6. Create and push annotated tag `v0.5.0`. Wait for the protected release
   environment, then inspect the draft: DMG, EXE, MSI, updater archives, `.sig`
   files, `latest.json`, per-platform checksums, and provenance.
7. Verify checksums and `latest.json` platform keys before publishing the draft.
   Publish only when every required asset is present and the tag/main commit
   relationship is exact.
8. Download and install the macOS artifact, preserve the existing application
   data, launch the installed app, and exercise the representative workflows.
9. Verify the live docs/download page, GitHub `releases/latest`, updater metadata,
   pack compatibility, and final clean tracked tree.

Verdict must be `PASS` before completion. `PASS WITH FOLLOW-UPS` is acceptable
only for the already-disclosed unpaid OS code-signing limitation; any functional,
privacy, updater, artifact, or exact-SHA gap is `HOLD` or `BLOCKED`.
